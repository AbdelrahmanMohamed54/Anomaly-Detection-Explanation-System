# Project: AI Anomaly Detection & Explanation System

## What this project does
A hybrid AI system combining a TensorFlow time-series anomaly detection model
with a LangChain LLM agent that automatically generates plain-language root
cause analysis (RCA) reports when production anomalies are flagged.
Reports delivered in English and German via a Grafana-integrated dashboard.

## Architecture
Sensor Data (time-series CSV / PostgreSQL)
    → data_preprocessing.py (feature engineering, normalization)
    → TensorFlow anomaly_detector.py (LSTM/Autoencoder — flags anomalies)
    → LangChain RCA Agent (rca_agent.py — root cause analysis)
        → Tool 1: historical_query.py (PostgreSQL — past similar anomalies)
        → Tool 2: semantic_search.py (ChromaDB — incident report retrieval)
        → Tool 3: action_recommender.py (ChromaDB — maintenance log RAG)
    → report_generator.py (bilingual EN/DE structured report)
    → FastAPI backend (api/main.py)
    → Grafana dashboard (real-time alerts + LLM report panel)

## Tech stack
- Anomaly detection: TensorFlow (LSTM autoencoder on time-series data)
- LLM agent: LangChain + Google Gemini 2.5 Flash (primary)
- LLM fallback: Groq Llama 3.3 70B
- Historical DB: PostgreSQL via SQLAlchemy + psycopg2
- Semantic search: ChromaDB + sentence-transformers (all-MiniLM-L6-v2)
- API: FastAPI
- Dashboard: Grafana (importable dashboard.json)
- Containers: Docker + Docker Compose

## Pydantic models (define in api/main.py)
- SensorReading: sensor_id, timestamp, values: dict[str, float]
- AnomalyEvent: sensor_id, anomaly_type, severity: float, timestamp, detected_values
- HistoricalAnomaly: id, sensor_id, anomaly_type, timestamp, duration, resolution
- RCAReport: sensor_id, severity, anomaly_summary, root_cause, similar_incidents,
             recommended_actions, escalate: bool, sources: list[str],
             generation_time_seconds: float, language: str

## TensorFlow model design
- Architecture: LSTM Autoencoder (encoder-decoder)
- Input: sliding window of 50 time steps, multivariate sensor readings
- Output: reconstruction error per time step
- Anomaly threshold: mean + 3*std of reconstruction error on normal training data
- Training data: synthetic normal sensor data generated by generate_synthetic_data.py
- Save/load: model/saved_models/ directory using model.save() / tf.keras.models.load_model()

## Key design decisions
- Synthetic data first: generate_synthetic_data.py creates realistic sensor patterns
  so the model can be trained without real proprietary data
- Three independent ChromaDB collections: incident_reports, maintenance_logs (separate)
- PostgreSQL for structured historical queries, ChromaDB for semantic search
- Bilingual output: all reports generated in both EN and DE in one API call
- Grafana reads from FastAPI via JSON data source plugin (no direct DB connection)
- Provider-agnostic LLM: swap Gemini for Groq by changing one environment variable

## Commands
- Generate synthetic data: python scripts/generate_synthetic_data.py
- Seed PostgreSQL: python scripts/seed_postgres.py
- Ingest incident reports: python scripts/ingest_incident_reports.py
- Ingest maintenance logs: python scripts/ingest_maintenance_logs.py
- Train model: python model/train.py
- Start API: uvicorn api.main:app --reload --port 8000
- Run tests: pytest tests/
- Full Docker stack: docker-compose up --build

## Known limitations
- PostgreSQL must be running for historical_anomaly_tool to return results;
  falls back gracefully to empty list when unavailable
- Docker test not run in dev environment (daemon not available);
  all code verified via 64 unit/integration tests
- German translation quality depends on LLM; fallback returns EN report with 'de' tag
- In-memory history resets on API restart; no persistence layer
